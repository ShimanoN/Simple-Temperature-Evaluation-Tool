## 📌 既知制限事項（v1.0.0）

### 1. タイムスタンプ機能なし（重要度: 中）

#### 現状
- CSV に記録される時刻は **相対時刻（経過秒）のみ**
- 例：`ElapsedSec,Temp_C,State,...` の ElapsedSec は RUN開始からの秒数

#### 制限の理由
- M5Stack本体に **RTC（リアルタイムクロック）がない**
- RTC追加には I2C デバイス＆バッテリー必要
- 引き継ぎ時間（1ヶ月）の制約

#### 影響
- CSV ファイルだけからは「いつ計測したのか」が不明
- 複数ラン・複数日間のデータを統合分析する際、日時軸が不備
- **回避策**: ファイル名やメタデータ（外部記録）で日時を管理

#### 改善案（Phase 5推定4週間）
```
【実装内容】
- DS3231 等の I2C RTC デバイス追加
- CSV ヘッダに新列: Timestamp_YYYYMMDD_HHMMSS
- M5Stack IDLE 画面に日時表示
- ボタンで手動時刻設定UI

【工数】
- RTC ドライバ実装: 2日
- CSV フォーマット拡張: 1日
- UI 実装: 1.5日
- テスト・ドキュメント: 1.5日
→ 合計 6日
```

---

### 2. クラウド連携なし（重要度: 低～中）

#### 現状
- 計測データは **M5Stack 内のMicroSD のみに保存**
- クラウドストレージ・リモート監視機能なし

#### 制限の理由
- WiFi 認証・暗号通信の実装が複雑
- AWS IoT / Azure IoT Hub 連携は別途認証設定・ライブラリ統合が必要
- 引き継ぎ時間の制約

#### 影響
- 複数の計測地点からリアルタイムにデータを集約できない
- 歴史的レポート作成にはMicroSDを毎回回収する必要
- リモートアラーム（メール通知等）ができない

#### 改善案（Phase 6推定2週間）
```
【実装内容】
- Arduino WiFi ライブラリ統合
- AWS IoT Core コネクター
- MQTT publish 実装
- オフライン時ローカルキューイング

【工数】
- WiFi コネクト: 2日
- AWS IoT SDK 統合: 2日
- MQTT 実装: 1.5日
- テスト・ドキュメント: 1.5日
→ 合計 7日
```

---

### 3. LCD グラフ表示なし（重要度: 低）

#### 現状
- LCD は**テキスト情報のみ**（温度値・アラーム状態など）
- グラフはMicroSD → PC→ Excel で作成

#### 制限の理由
- M5Stack Core2 の320×240解像度は低く、グラフ描画ロジックが複雑
- リアルタイムグラフは描画性能が問題（毎秒更新の場合）
- 引き継ぎ時間の制約

#### 影響
- 計測中にトレンドビジュアルが見えない
- PC なしでは計測結果を視覚化できない

#### 改善案（Phase 7推定3日）
```
【実装内容】
- TFT Canvas ライブラリ活用
- 簡易折れ線グラフ描画（最直近100点の表示）
- LUT（ルックアップテーブル）キャッシュで描画性能向上

【工数】
- グラフ描画ロジック: 1.5日
- UX 設計・テスト: 1.5日
→ 合計 3日
```

---

### 4. 単一センサのみ対応（重要度: 中）

#### 現状
- K型熱電対 + MAX31855 の **シングルチャネルのみ**
- 複数箇所の同時計測ができない

#### 制限の理由
- M5Stack の I2C / SPI ポートは限定
- ハードウェア設計時点で単一センサ想定
- I2C Multiplexer や複数 MAX31855 の制御ロジックが別途必要

#### 影響
- 複数点の温度（例：チャンバー内の複数カ所）を同時計測できない
- 異なる液体・物体の温度を同時に記録できない

#### 改善案（Phase 8推定2週間）
```
【実装内容】
- I2C 4チャネル Multiplexer (TCA9548A等) 導入
- MAX31855 ×4 並列接続
- CSV 列拡張: Temp_C_CH0, Temp_C_CH1, ...
- UI マルチチャネル表示

【工数】
- ハードウェア設計・配線: 1.5日
- I2C Mux ドライバ: 1日
- 計測ロジック拡張: 1.5日
- CSV フォーマット拡張: 1日
- UI 実装・テスト: 2日
→ 合計 7日
```

---

### 5. USB給電のみ（重要度: 低）

#### 現状
- **USB-C 給電のみ対応**
- バッテリー駆動不可
- ケーブル長制約あり

#### 制限の理由
- リチウム電池＋充電管理IC の追加が必要
- 過充電・過放電保護回路の設計が別途必要
- 引き継ぎ時間の制約

#### 影響
- 現場で長時間の計測ができない（ケーブル範囲内のみ）
- 移動計測が困難

#### 改善案（Phase 9推定2週間）
```
【実装内容】
- 3000mAh リチウムポリマー電池 搭載
- TP4065 USB充電管理IC
- LCD に電池残量表示
- 低電力モード実装

【工数】
- バッテリーIC 設計・焼き込み: 2日
- 電源管理ドライバ: 1.5日
- UI・テスト: 1.5日
→ 合計 5日
```

## 🔬 温度計測に関する仕様の理解

### 計測精度の限界

```
MAX31855 仕様:
  - 分解能: 0.25℃ (12-bit)
  - 誤差範囲: ±2℃程度
  
実測 (K型熱電対):
  - K型の内部誤差: ±1℃相当
  - MAX31855 量子化誤差: ±0.125℃
  
実現精度:
  → ±1℃程度 (相対精度は良い)
```

### 応答時間

```
熱電対→MAX31855→メモリ の流れ:
  1. 熱電対応答: ≈100ms (物理応答)
  2. MAX31855 I2C読取: ≈10ms
  3. フィルター処理: ≈0ms (Welford原理上リアルタイム)
  
→ 総応答時間: ≈100ms
  （急急な温度変化は100ms遅れて観測される）
```

### 温度範囲の制限

```
K型熱電対理論値:
  - 計測下限: -50℃
  - 計測上限: +150℃
  
MAX31855 拡張範囲:
  - 実装回路側の耐熱性で-40～+85℃推奨
  
実運用:
  - 推奨範囲: -20℃～+120℃
  - X外は計測不可 or 誤差激増
```

## 📚 CSV フォーマットの固定化

### 現状（v1.0.0）

```
ElapsedSec,Temp_C,State,Samples,Average_C,StdDev_C,Max_C,Min_C,HI_ALARM,LO_ALARM
```

### 設計理由

- **シンプル**: 初期段階での十分な情報量
- **Excel互換**: 直接開いて分析可能
- **拡張性**: 新列を末尾に追加可能（後方互換性維持）

### 将来の拡張案（後方互換性あり）

```
Phase 5 (RTC統合):
  → Timestamp_YYYYMMDD_HHMMSS 列を最後に追加

Phase 8 (複数センサ):
  → Temp_C_CH1, Temp_C_CH2, ... を最後に追加
  （既存の Temp_C は単一センサモードのままにしておく）
```

## 🔧 開発者向け：コード変更時の注意事項

### 後方互換性について

新機能追加時は、既存ユーザのCSV解析スクリプトを壊さないこと：

**❌ 悪い例**:
```cpp
// Temp_C 列をリネームしてしまう
"Temperature_Celsius,..." // 既存スクリプト破損
```

**✅ 良い例**:
```cpp
// 新しい列は末尾に追加
"Temp_C,...,Timestamp_YYYYMMDD_HHMMSS"  // 既存スクリプトそのまま動く
```

### テスト追加時

新しい機能は Phase 4 統合テストテンプレートに従う：

```markdown
## 【N】新機能テスト

### N-01: 新機能基本動作
| 項目 | 内容 |
|:---|:---|
| テスト ID | N-01 |
| テスト名 | 新機能の基本的な動作確認 |
| ...
```
