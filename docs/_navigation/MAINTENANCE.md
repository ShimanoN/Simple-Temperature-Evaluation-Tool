# ドキュメント保守ガイド

**目標**: ドキュメント保守を継続的かつ実行的に取り組む方法  
**対象**: ドキュメント管理・セッション終了時に実行  
**最終更新**: 2026年2月26日  

---

## Ⅰ ドキュメント保守の基本原則

### **原則1：説明できる範囲の情報精度**

各トピックについて、複数のファイルで同じ内容が存在しないようにします。また説明が必要な場合は、「**参照リンク**」で指示します。

```markdown
✅ 正しい場合
- SETUP_GUIDE.md に「初期セットアップ手順」記載
- QUICKSTART.md にも「初期セットアップ手順」記載

❌ 間違い
- QUICKSTART.md：「詳細は [SETUP_GUIDE.md](SETUP_GUIDE.md) を参照」
```

### **原則2：最新情報の保証**

コードに変更があった場合、ドキュメントも同時に更新します。更新日付が古いままだと、内容が古い状態のまま使用されます。

### **原則3：新規開発者への説明**

新しいメンバーがこのドキュメントで分かるかどうかを独自に判断します。必要に応じて解説内容を付け足します。

### **原則4：段階的な詳細度**

- README.md：全体概観（5分）
- 詳細書：基本的な知識（15～20分）
- 詳細書：深掘り（30分以上）

読者が「必要な範囲」の情報に最善でアクセスできる設定が重要です。

## Ⅱ セッション終了時のドキュメント更新フロー

各セッション終了時に実行してください。総所要時間：30～45分

### **Step 1：コード変更のドキュメント調査**（15分）

**対象ファイル**（優先度順）：
1. **CODE_EXPLANATION.md** - 最優先
   - 変更したアルゴリズムの説明を更新
   - 新しい関数が追加されたら、そのセクションを追加
   - Stage 別の実装内容を確認

2. **CODE_REFERENCE.md**
   - API 変更があれば、関数形情報を更新
   - パラメータ追加／削除があれば修正
   - Task 名変更があれば対応

3. **detailed_spec_sw.md**
   - アルゴリズムの大幅な変更があれば、詳細を更新
   - セットアップ手順が変わったら修正

**チェックリスト**：
- [ ] 新しい関数・クラスが CODE_EXPLANATION.md に記載されているか
- [ ] パラメータ変更が CODE_REFERENCE.md に反映されているか
- [ ] 削除された概念・関数など誤りがないか
- [ ] 最終更新日付が本日の日付になっているか

### **Step 2：セッション完了報告書の作成**（20分）

**SESSION[N]_COMPLETION_REPORT.md** を作成

```markdown
# Session [N] 完了報告書

## 完了内容
- Task 1：[内容] → ✅
- Task 2：[内容] → ✅

## ビルド成功状態
- ✅ SUCCESS

## 次ステップ
- [ ] ...

---

**最終更新**: [YYYY年MM月D日]
```

**記載内容**：
- 完了したタスク一覧（チェックボックス付き）
- コード行数・関数数
- ビルド成功状態、フラッシュ率、メモリ使用率
- 謚陦鍋情・Learning
- 次ステップへの指摘事項

### **Step 3：README.md と INDEX.md の更新**（10分）

**README.md**：
- 「現在のプロジェクト概況」セクションを更新
- 最新セッション完了報告書へのリンクを修正
- 更新日付に最新のセッション情報を追加

**INDEX.md**：
- カテゴリー別のファイル説明文を最新化
- 新しいドキュメントがあれば追加
- 廃棄ドキュメントの説明を補正

**チェックリスト**：
- [ ] README.md の「現在のプロジェクト概況」は最新か
- [ ] INDEX.md の「ファイル別詳細情報」に新規ファイルが記載されているか
- [ ] リンク破れがないか

### **Step 4：廃棄ドキュメントの処理**（5分）

3ヶ月以上更新されていないドキュメント、または内容が古いドキュメントについて

| パターン | 対応 |
|---------|------|
| 参考情報あり | README / INDEX で「参照方法」と指示 |
| 内容が古くなった | 削除 or 内容を最新情報に統合 |
| 一部が完了済み | 廃棄マーク削除 + リンク修正 |

**例**：
```markdown
✅ REFACTORING_PLAN.md
   • Phase 2-B はすでに完了。参照資料のみ
   • STAGE_4_REFACTORING_HANDOVER.md を参照
```

## Ⅲ 継続的なドキュメント監査

### **定期実行**（30分）

```bash

1. 最終更新日付が3ヶ月以上古いか?
2. リンク破れがないか?
3. コードが変わったのにドキュメント未更新か?
```

### **半年ごとの完了**（2時間）

```bash
# ドキュメント保守の最優先確認
1. 古いファイル・内容がないか?
2. 読者ごとにドキュメント体験は最適か?
3. 新規開発者が容易に学習可能か?
4. README → INDEX → 詳細書 への誘導は明確か?
```

## Ⅴ ドキュメント評価の三段階の指標

### **蛛・肩な状況**✅

- README.md が最初のエントリーポイント
- 各ドキュメントに目的・対象読者が明記
- 最終更新日が3ヶ月以内
- リンク破れない
- 新規開発者が「どれを読むべき」に指示

### **注意が必要な状況**⚠️

- 最終更新日が3ヶ月以上経過
- リンク破れが複数ある
- 同じ内容が複数ファイルに存在
- 新規開発者が辺り構見で情報を走破

### **蜊ｱ讖溽噪な状況**🔴

- 最終更新日付が3ヶ月以上経過
- コードとドキュメントがズレている
- ドキュメント間に矛盾がある
- 新規開発者が「このドキュメント使用できるか」と獴安

## Ⅵ ドキュメント間の参照ルール

### **正しい参照方法**

```markdown
✅ [CODE_EXPLANATION.md](CODE_EXPLANATION.md#アルゴリズム別詳細)
✅ [詳細は こちら](CODE_EXPLANATION.md)
```

### **避けるべき参照**

```markdown
❌ "CODE_EXPLANATION参照"  （ファイル名だけ）
❌ [参照](../docs/CODE_EXPLANATION.md)  (相対パス長重)
❌ [CODE_EXPLANATION.md](./../code/CODE_EXPLANATION.md)  
   （無駄なパスコンポーネント含有）
```

### **参照先ドキュメントが削除された場合**

```markdown
❌ 削除されたファイル: OLD_SPEC.md
✅ 修正: 詳細内容を CODE_EXPLANATION.md に統合
      参照していたドキュメントを全て更新
```

## Ⅸ ドキュメント品質向上の推奨

ドキュメント・削除・改善・段階帰納**で求められます

### **フィードバック受信**

1. 新規開発者に読ませて、「ここが分かりずらい」をリサーチ
2. 「このドキュメントの説明に正った」の容詞を記録
3. 詐期的に改善・追加をセッション終了に含み込み

### **改善・プロセス**

```
フィードバック受信
  ↓
改善方法をタスクリストに追加
  ↓
セッション中で実装
  ↓
完了報告書に記載
  ↓
ドキュメント更新
```

## Ⅹ 優先：ドキュメント・コード・同等性

> **「コード・短期的な宣言、ドキュメント・長期的な宣言」**

ドキュメント・を保守する事は、以下を達成します

- ✅ 新規開発者・ on-boarding 時間を 50% 短縮
- ✅ バグ調査時に 滞留・ロスを削減
- ✅ プロジェクト成長・財務継続を保証
- ✅ 保守時に資料・参照・を迅速に行える
- ✅ 自分で書いた・ても・数ヶ月後に「何をしている」の理解・を短縮

**ドキュメント保守 = プロジェクト保証**
