## 2. 最重要アピールポイント：PLC→C++ 技術転用ストーリー

これが他の候補者と最大限に差別化できる「唯一無二のストーリーライン」である。

### 2-1. PLC思考とC++の対比マッピング

| PLC の概念 | C++ での実装 | 転用の根拠 |
|:---|:---|:---|
| スキャン周期（固定周期実行） | `millis()`ベースの3タスク分割（10ms/50ms/200ms） | 「一定周期で処理を回す」制御思考は同じ |
| グローバルラベル（全POU共有変数） | `struct G { ... }` 一元管理構造体 | 「全モジュールが参照する変数はひとつの場所に集める」設計思想の転用 |
| 工程歩進（S0→S1→S2フロー） | State Machine（IDLE→RUN→RESULT） | 「今どの工程にいるかで動作が変わる」という状態管理の本質は同じ |
| インターロック条件 | アラーム判定ロジック（HI/LO_ALARM） | 「条件を満たした時だけ次へ進む」安全制御の思考の転用 |

### 2-2. このストーリーが示すもの

- **技術の本質理解**: 特定言語や環境への依存ではなく、制御工学的な思考パターンを持っている
- **学習効率**: 既存知識を足場にして新領域に入るため、ゼロからの学習より大幅に速い
- **汎用性**: 「PLC→C++」で効いたパターンは「C++→別言語」「組み込み→クラウド」にも応用可能

## 4. 正直な文脈設定（過大表現を避けるガイド）

### 書いてよいこと

- 「16,082円で専用計測ツールを完成させた」（事実）
- 「33ケースの統合テストを設計・実行し100%通過を確認した」（事実）
- 「PLC経験をC++設計に転用する独自の学習アプローチを実践した」（事実）
- 「GitHub公開＋引き継ぎドキュメント整備で内外に公開できる状態で完成させた」（事実）

### 書いてはいけないこと・注意が必要なこと

- 「現場の工数を削減した」→ 現場実証未完了。「～を削減することを目的として開発した」が正確
- 「高精度温度計測」→ ±1℃精度はエントリーレベル。「実用精度の温度計測」が適切
- 「マルチタスク処理」→ RTOSなし。「時分割処理（マルチレート制御）」が正確
- 「業務で使用中」→ 未実証。「業務適用を想定した試作ツール」が正確

## 5. 上司向け報告書 文章スタイル指示

### 構造テンプレート（BOSS_REPORT に使用）

```
【概要1文】何をいくらで、どのくらいの期間で作ったか
【背景】なぜ必要だったのか（課題）
【成果】何ができるようになったか（ビジネス価値）
【技術品質】品質担保の証拠（テスト100%、コスト、安全性）
【今後の可能性】次のステップがあること（Phase 5〜9）
【結論】組織への貢献可能性
```

### 文体

- 丁寧体（です・ます調）
- 技術用語は初出時のみかっこ書きで補足（例：統計計算（平均・標準偏差））
- 箇条書きを多用し、文章の密度を下げる
- A4 1枚〜1.5枚を目安（長すぎない）

## 6. 職務経歴書 文章スタイル指示

### STAR 法（Situation / Task / Action / Result）に沿った記述

```
【Situation】どんな業務環境・課題の中で
【Task】どんな目標・要件を設定し
【Action】どんな設計判断・実装・デバッグを行い
【Result】どんな測定可能な成果が得られたか

### 技術キーワード（プロフィール・スキルセクションに記載すべき）

```
C++ / ESP32 / Arduino framework / PlatformIO / SPI / I2C
Welford法（オンライン統計） / 状態機械設計 / マルチレートタスク制御
MicroSD(FAT32) / EEPROM / MAX31855 / Git/GitHub
統合テスト計画策定 / 技術ドキュメント整備 / PLC→C++ 技術転用

### 職歴書上の位置づけ

- カテゴリ: 「個人開発プロジェクト（自主業務・スキルアップ）」
- 期間: 2026年2月〜3月
- 主要技術: C++ / ESP32 / PlatformIO
- プロジェクト規模: ソースコード約1,400行 / ドキュメント15ファイル / テスト33ケース

## 7. Haiku 向け 出力品質管理ルール

（Task 3・Task 4 の Haiku に必ず守らせること）

### 必須ルール

1. **数値は原文から引用**: 記憶や推測で数値を書かない。RAW_ACHIEVEMENTS.md の数値テーブルのみ参照
2. **[TODO: Sonnet確認]タグ**: 「感情」「評価」「比較優位」など主観表現が入る箇所にはタグを付ける
3. **事実と解釈を分離**: 事実文と解釈文を混在させない（「○○を実装した（事実）。これにより～の課題を解決した（解釈）」と明示分離）
4. **過去形の徹底**: 職務経歴書はすべて完了形で記載（「現在進行中」の表現は使わない）
5. **省略禁止**: 「詳細は省略」「記載省略」等のプレースホルダーなし。不明なことは「[要確認: ○○]」と明示

### 品質チェック観点（Sonnet が最終版を作成する前の Haiku 評価基準）

| チェック項目 | OK | NG |
|:---|:---|:---|
| 数値の正確性 | RAW_ACHIEVEMENTS.mdの数値と一致 | 四捨五入・近似値・不明な数値 |
| 誇張表現 | 「業務適用を想定した試作ツール」 | 「現場で活躍中のツール」 |
| 技術語の精度 | 「時分割タスク制御」 | 「マルチスレッド」「並列処理」 |
| ストーリーの一貫性 | PLCアナロジーが自然に流れる | 唐突なPLC言及 |
