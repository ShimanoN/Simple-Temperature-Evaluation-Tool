## 2. 担当業務

### 2-1. ハードウェア企画・調達・組み立て

- 業務課題の分析：ヒートガン/チャンバーの温度評価におけるデータ記録工数の課題を特定
- 部品選定：M5Stack Core2/CoreS3（ESP32）+ K型熱電対 + MAX31855 SPI/I2C変換ICの組み合わせを採用
  **根拠**: コスト（16,082円以内）・実装難易度・将来拡張性のバランス評価
- 全パーツ調達：5部品を予算内で確保
- ハンダ付け不要設計：ジャンパワイヤ＋完成品モジュール組み合わせで、容易な組み立てを実現

### 2-2. ソフトウェアアーキテクチャ設計

#### マルチレートタスク分割設計（3タスク構成）

- **IO_Task（10ms周期）**: センサ読取（MAX31855）/ ローパスフィルタ処理 / SD書き込み判定
- **Logic_Task（50ms周期）**: Welford法統計計算 / アラーム判定 / 状態遷移制御
- **UI_Task（200ms周期）**: LCD描画更新 / ボタン入力処理

**設計根拠**: PLC的「スキャン周期」の思考をC++マルチタスク実装に転用。高頻度（温度読取）と低頻度（画面描画）の干渉を排除

#### グローバル変数管理

- `struct G { ... }` による全変数の一元管理パターンを採用
- **転用元**: PLC的「グローバルラベル」の概念。全POUが参照する変数をひとつの場所に集約

#### 状態機械設計

- IDLE（待機・センサ表示）→ RUN（計測中・リアルタイム更新）→ RESULT（結果表示・CSV保存）→ IDLE の循環
- **設計根拠**: PLC工程歩進（S0→S1→S2）の思考パターン。「今どの状態か」を明確化し、誤動作を防止

### 2-3. アルゴリズム実装

#### Welford法（オンライン統計計算）

- **用途**: 平均・分散・標準偏差をデータ蓄積なしでリアルタイム更新
- **実装**: D_Sum / D_M2 による逐次更新式をLogic_Task内で50ms周期で計算
- **利点**: メモリ使用量が計測時間に依存しない（RAM 7.2% の低使用率を実現）

#### ローパスフィルタ（一次遅れフィルタ、α=0.1）

- **用途**: MAX31855センサ読取値のノイズ低減
- **実装**: 前回値90% + 新規値10% の加重平均
- **チューニング**: 実機確認とCSVデータ分析により係数α=0.1を決定

### 2-4. デバッグ（5件の技術課題解決）

#### Bug-01: RUN中のAverage/StdDevがすべて0.0のままになる現象

| 項目 | 内容 |
|:---|:---|
| **発見** | CSVファイル目視で Average_C列が全行0.0 |
| **仮説** | 書き込み条件が早すぎる（`D_Count >= 2`） |
| **根本原因追跡** | IO_Task(10ms) がLogic_Task(50ms)より先に実行 → Welford未実行で0.0が記録 |
| **修正** | 書き込み条件を `D_Count >= 10`（100ms待機）に変更 → Logic_Taskが最低2回完了保証 |

#### Bug-02: Welford計算されているのにD_Averageが0.0のまま（複合バグ）

| 項目 | 内容 |
|:---|:---|
| **発見** | Bug-01修正後の再テストで0.0継続 → コード精読 |
| **症状** | D_Sum/D_M2は更新されるがD_Average=NAN |
| **根本原因** | D_Average計算が RESULT遷移時のみ → タイミングギャップ |
| **修正** | Logic_Task内で毎回計算（50ms周期）に変更 |

#### Bug-03: 温度読取の一時的異常値（スパイク）

- **症状**: MAX31855 の readCelsius() が稀に誤値を返す
- **原因**: SPI通信の一時的な乱れ
- **修正**: 異常値検出時に最大3回リトライ処理を実装

#### Bug-04: SD書き込み後のSPIバス競合

- **症状**: SD書き込み後に MAX31855読取値が不定になる
- **原因**: SD（SPI）書き込み後にバス状態が不定 → MAX31855と共用SPI干渉
- **修正**: 書き込み後に `SPI.end(); SPI.begin();` でバスリセット

#### Bug-05: MicroSD初期化失敗

- **症状**: 特定条件でSDカード初期化ができない
- **原因**: CS ピン未指定（GPIO4 が明示されていない）
- **修正**: `SD.begin(4)` として microSD CS = GPIO4 に明示化

## 3. 使用技術・ツール

### 3-1. プログラミング言語とフレームワーク

- **C++** (ESP32向けArduino フレームワーク)
- **型安全**: constexpr定数化、型サフィックス管理、NaNガード
- **構造体設計**: struct G { ... } による変数一元管理
- **エラーハンドリング**: リトライ処理、ゼロ除算保護、モード別エラー表示

### 3-2. マイコン・ハードウェア

- **マイコン**: ESP32（M5Stack Core2 / CoreS3上で実行）
- **温度センサ**: K型熱電対 ＋ MAX31855 SPI/I2Cデジタル変換IC
- **ストレージ**:
  - MicroSD（SPI接続、FAT32形式、CSV出力）
  - 内蔵EEPROM（I2C接続、設定デ永続保存）
- **GPIO制御**: ボタン入力処理（デバウンス実装）、ピン割り当て管理

### 3-3. 通信プロトコル

- **SPI**: MAX31855 & MicroSD の2デバイス同時接続（CS切り替え管理）
- **I2C**: EEPROM アクセス（設定値読み書き）

### 3-4. 開発環境

- **PlatformIO**: ビルド管理・ライブラリ管理
- **VSCode**: エディタ・デバッグ環境
- **Git / GitHub**: バージョン管理・公開リポジトリ（MIT License、v1.0.0 タグ）
- **シリアルモニタ**: 非侵襲デバッグ（ロジックアナライザなし）

### 3-5. アルゴリズム・設計パターン

- **Welford法**: オンライン平均・分散・標準偏差計算
- **ローパスフィルタ** (一次遅れ、α=0.1): ノイズ低減
- **状態機械**: enum class + switch による IDLE/RUN/RESULT の状態遷移
- **マルチレートタスク制御**: millis()タイマーによる10ms/50ms/200msの非同期実行
- **バッファリング**: 10サンプル毎のSD書き込み（都度書き込みより高速化）

## 5. 課題と対応（Problem / Action / Result 形式）

### Case 1: 計測データ自動記録の実現

**Problem**  
現場での温度評価時に、計測値の手書き記録またはExcel手入力が必要。評価1回あたりの工数が大きい。

**Action**  
MicroSD + FAT32ファイルシステム + CSVフォーマットの組み合わせを採用。計測中は10サンプル毎にSD書き込みを行い、バッファリングにより書き込み負荷を最適化。

**Result**  
計測終了時に DATA_NNNN.csv ファイルが自動生成され、PCのExcelで直接読み込み・グラフ化が可能に。手入力工数をゼロ化。

---

### Case 2: 統計値（平均・標準偏差）のリアルタイム計算

**Problem**  
統計値が計測終了後の手計算または別ツール計算に依存。計測中にばらつきトレンドを把握できない。

**Action**  
Welford法（オンライン統計計算）をLogic_Task（50ms周期）内で実装。D_Sum / D_M2 の逐次更新により、データ蓄積なしでリアルタイム更新を実現。

**Result**  
計測中のRESULT状態で平均・標準偏差・最大・最小が画面表示される。統計計算の外部依存を排除し、意思決定の即時性を向上。

---

### Case 3: センサノイズ対策と異常値排除

**Problem**  
MAX31855からの読取値が稀に異常値（スパイク）を返す。CSVに混入すると統計値の精度が低下。

**Action**  
- ローパスフィルタ（α=0.1）によるノイズ低減
- MAX31855読取時に異常値検出→最大3回リトライ処理を実装

**Result**  
スパイク除外率 > 99%。統計値の信頼性が向上し、計測精度 ±1℃を安定的に維持できるように。

---

### Case 4: SPI共有バスの競合排除

**Problem**  
MicroSD書き込み後にMAX31855読取値が不定になる現象。特定条件で計測値の信頼性が崩壊。

**Action**  
SPI.end() / SPI.begin() によるSPIバスリセットをSD書き込み後に挿入。バス状態の確実なクリアを実現。

**Result**  
SD書き込みとセンサ読取の間に干渉が生じなくなり。計測の全期間において安定した読取値を確保。

---

### Case 5: テストによる品質保証

**Problem**  
複数のバグが混在していたため、個別デバッグだけでは見落としリスクが高い。リリース基準が不明確。

**Action**  
7カテゴリ33ケースのテスト計画書を自作。テスト実行→問題発見→修正→再テストのループを徹底。

**Result**  
33/33 テスト合格（100%）を確認。GitHub公開・引き継ぎ実装可能な「リリースレベル」の品質を達成。
