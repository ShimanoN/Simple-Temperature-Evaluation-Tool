## 【職務要約（冒頭サマリー用）】

PLC（プログラマブルロジックコントローラ）を用いた制御設計の経験を基盤に、C++組み込み開発をゼロから習得し、自主開発プロジェクトとして**ESP32マイコンを使用した温度計測・評価ツール**を約1ヶ月で設計・実装・テスト・ドキュメント化まで一貫して完遂しました。

PLC設計で培った「周期制御」「工程歩進（ステートマシン）」「グローバル変数管理」の思考パターンをC++に転用することで、効率的な技術習得と設計品質の両立を実現しました。企画から公開まで一人でやりきった本プロジェクトは、自走力・問題解決力・継続的な技術習得能力の証明です。

### 開発背景（Situation）

現場での温度評価作業（ヒートガン・恒温槽・チャンバーを使用した評価）において、計測データの手書き記録・事後の統計集計・リアルタイムの異常温度検知が困難という業務課題が存在していました。汎用の温度計では計測と記録が連動せず、評価1回あたりに要する工数が大きい状況でした。

---

### 担った役割と目標（Task）

専用の温度計測ツールを低コスト（2万円以内）かつ現場で一人で使えるシンプルな操作性で実現することを目標として設定しました。技術的な要件として、C++による組み込み開発（自身にとって本格的な習得は初）、SPI/I2C通信の実装、Welford法によるオンライン統計計算、MicroSDへのCSV出力を自力で実装することが必要でした。

---

### 実施した技術的アクション（Action）

#### A-1. ソフトウェアアーキテクチャ設計（PLC経験の転用）

PLC設計で培った概念をC++に体系的に転用する設計アプローチを採用しました。

| PLC の設計手法 | C++ での実装 | 転用の意図 |
|:---|:---|:---|
| スキャン周期（固定周期処理） | `millis()` による3タスクの時分割制御（IO:10ms / Logic:50ms / UI:200ms） | センサ読取と画面描画の干渉を防ぐため周期を分離 |
| グローバルラベル（全局共有変数） | `struct G { ... }` による全変数の一元管理 | モジュール間の変数参照を一箇所に集約 |
| 工程歩進（S0→S1→S2） | State Machine（IDLE→RUN→RESULT→IDLE） | 「今どの状態か」に応じて処理を切り替え、誤動作を防止 |
| インターロック条件 | HI/LOアラーム判定ロジック | 設定温度超過のリアルタイム検知 |

#### A-2. アルゴリズム実装

**Welford法（オンライン統計計算）の実装**  
計測データを配列に蓄積せず、逐次的に平均・分散・標準偏差を更新するアルゴリズム。長時間計測でもRAM使用量が増加しないことを設計段階で確認し、採用を決定。

**一次遅れローパスフィルタ（α=0.1）の実装**  
センサ読取値のノイズを低減するため、前回値を90%・新規値を10%の重みで加重平均。定数α=0.1は実機での目視確認とCSVデータ分析によりチューニング。

#### A-3. 技術課題の自力デバッグ（5件）

ロジックアナライザなし、シリアルモニタ＋CSV目視での診断を実施しました。代表的なデバッグ事例を以下に示します。

**事例1: 統計値がすべて0.0のまま（複合バグ）**

```
【症状の発見】: CSVファイルのAverage_C列が全行0.0
【仮説】: 書き込みタイミングが早すぎる？
【調査】: IO_Task(10ms)がLogic_Task(50ms)より先に実行されていることを確認
【根本原因（Bug-01）】: D_Count>=2で書き込み → Welford未実行のまま0.0が記録される
【修正】: 書き込み条件をD_Count>=10（100ms待機）に変更
【再確認で別バグ発見（Bug-02）】: 0.0継続 → コード精読でRESULT遷移時のみ計算する設計バグを発見
【修正】: Logic_Task(50ms)内で毎回計算するよう変更
```

**事例2: SPI バスの競合**
SD書き込み後にMAX31855（温度センサ）の読取値が不定になる現象。SPI共用バスの状態不定が原因と特定し、SD書き込み後に`SPI.end(); SPI.begin();`でバスリセットを実装（Bug-04）。

#### A-4. テスト設計と実行

7カテゴリ33ケースの**統合テスト計画書**を自作しました（docs/PHASE4_INTEGRATION_TEST_PLAN.mdとして整備）。テストはすべて手動実施し、実機上での動作を確認。テスト中に発見した問題はその都度修正・再テストを行いました。

#### A-5. ドキュメント整備

引き継ぎを意識した技術ドキュメント15ファイルを整備しました（総行数 数千行）。

| ドキュメント | 内容 |
|:---|:---|
| IMPLEMENTATION_GUIDE.md | APIリファレンス・タスク設計仕様・CSVフォーマット詳細 |
| TROUBLESHOOTING.md | 7症状別のデバッグ手順書（722行） |
| FINAL_HANDOVER_GUIDE.md | 高校卒業1年目の後継者を想定した引き継ぎガイド（8パート構成） |
| CODE_EXPLANATION.md | 「なぜそう設計したか」の意図・思考プロセスを記録した解説書 |
| INTEGRATION_TEST_RESULTS.md | 33ケース全テスト結果・バグ修正履歴 |

## 【自己 PR ポイント（面接・エントリーシート用素材）】

### ポイント① ── 技術転用力（既存知識の応用）

PLC設計経験で習得した「スキャン周期」「工程歩進」「グローバルラベル」という制御工学の本質的な概念を、C++の「マルチレートタスク」「状態機械」「構造体変数管理」に体系的に対応させました。特定の言語・ツールへの依存ではなく、「制御の本質」として理解しているため、異なる技術環境でも同様のアプローチが適用できます。

### ポイント② ── 自走力（計画→実行→完遂）

16,082円の予算確保・部品調達・組み立て・ソフトウェア開発・33ケースのテスト・15ファイルのドキュメント整備・GitHub公開まで、一切外部委託なしで完遂しました。「誰かに頼まれたから」ではなく、「課題を自分で発見して、自分で解決した」プロジェクトとして、自走型エンジニアとしての働き方を体現しています。

### ポイント③ ── デバッグ・問題解決力

5件の技術的バグを自力で根本原因まで追跡・修正しました。特に「Bug-01→Bug-02の思考連鎖（統計値が0.0の原因を追ったら複合バグだった）」は、表面的な症状に惑わされず根本원因を探ることの重要性を実証しています。ロジックアナライザなし、シリアルモニタとCSVファイル目視という制約下での診断力は、実際の開発現場でも応用可能なスキルです。

### ポイント④ ── ドキュメント文化（後工程を意識した開発）

「自分しかわからないコード」を作らないことを意識し、高校卒業1年目の後継者を想定した引き継ぎガイドを含む15ファイルのドキュメントを整備しました。「APIリファレンス」「トラブルシューティング手順書」「設計思想の解説書」「テスト計画書」という多角的なドキュメント体系は、チームでの開発でも通用するソフトウェアエンジニアリング実践の証拠です。
