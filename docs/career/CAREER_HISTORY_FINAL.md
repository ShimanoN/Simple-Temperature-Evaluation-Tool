PLC（プログラマブルロジックコントローラ）を用いた制御設計の経験を基盤に、C++組み込み開発をゼロから習得し、自主開発で**ESP32マイコンを使用した温度計測・評価ツール**を設計・実装・テスト・ドキュメント化まで一貫して完遂しました。他業務と並行しながら暦上約1ヶ月、実作業は約5日という凝縮した期間で仕上げました。

### 開発背景（Situation）

現場での温度評価作業（ヒートガン・恒温槽・チャンバーを使用した評価）において、計測データの手書き記録・事後の統計集計・リアルタイムの異常温度検知が困難という業務課題が存在していました。汎用の温度計では計測と記録が連動せず、評価1回あたりに要する工数が大きい状況でした。

---

### 担った役割と目標（Task）

専用の温度計測ツールを低コスト（2万円以内）かつ現場で一人で使えるシンプルな操作性で実現することを目標として設定しました。技術的な要件として、C++による組み込み開発（自身にとって本格的な習得は初）、SPI/I2C通信の実装、Welford法によるオンライン統計計算、MicroSDへのCSV出力を自力で実装することが必要でした。

### 実施した技術的アクション（Action）

#### A-1. ソフトウェアアーキテクチャ設計（PLC経験の転用）

PLC設計で培った概念をC++に体系的に転用する設計アプローチを採用しました。

| PLC の設計手法 | C++ での実装 | 転用の意図 |
|:---|:---|:---|
| スキャン周期（固定周期処理） | `millis()` による3タスクの時分割制御（IO:10ms / Logic:50ms / UI:200ms） | センサ読取と画面描画の干渉を防ぐため周期を分離 |
| グローバルラベル（全局共有変数） | `struct G { ... }` による全変数の一元管理 | モジュール間の変数参照を一箇所に集約 |
| 工程歩進（S0→S1→S2） | State Machine（IDLE→RUN→RESULT→IDLE） | 「今どの状態か」に応じて処理を切り替え、誤動作を防止 |
| インターロック条件 | HI/LOアラーム判定ロジック | 設定温度超過のリアルタイム検知 |

#### A-2. アルゴリズム実装

**Welford法（オンライン統計計算）の実装**  
計測データを配列に蓄積せず、逐次的に平均・分散・標準偏差を更新するアルゴリズム。長時間計測でもRAM使用量が増加しないことを設計段階で確認し、採用を決定。

**一次遅れローパスフィルタ（α=0.1）の実装**  
センサ読取値のノイズを低減するため、前回値を90%・新規値を10%の重みで加重平均。定数α=0.1は実機での目視確認とCSVデータ分析によりチューニング。

#### A-3. 技術課題の自力デバッグ（5件）

ロジックアナライザなし、シリアルモニタ＋CSVファイル目視という制約下で、全バグを根本原因まで追跡・修正しました。

**代表事例: 統計値が0.0のまま ─ 単純障害と見せかけた複合バグの特定**

表面的な症状は「CSVの Average_C 列が全行0.0」という単純なものでした。しかし段階的に追跡すると、2つの独立したバグが連鎖していることが判明しました。

| ステップ | 内容 |
|:---|:---|
| **①症状の発見** | CSVの Average_C 列が計測中ずっと 0.0 のまま |
| **②仮説立案** | 「書き込み条件が早すぎてWelfordが未実行」と仮説 |
| **③根本原因1（Bug-01）** | IO_Task(10ms) が Logic_Task(50ms) より先に実行 → Welford 未実行のまま 0.0 を書き込んでいた |
| **④修正1** | 書き込み条件を `D_Count >= 10`（100ms待機）に変更 |
| **⑤再テストで 0.0 継続** | 修正後もまだ 0.0 → 別の原因が存在すると判断してコード精読 |
| **⑥根本原因2（Bug-02）** | `D_Average = D_Sum / D_Count` の計算が RESULT 遷移時のみに実装されていた設計バグを発見 |
| **⑦修正2・完全解決** | Logic_Task（50ms周期）内で毎回計算するよう変更 |

この事例が示すのは「症状が再現しても根本原因は複数あり得る」という認識と、「修正後に必ず再検証する」という習慣です。表面の症状だけで判断せず、設計の意図まで遡って検証するデバッグアプローチを実践しています。

**その他のデバッグ事例（4件）**

| バグ | 症状 | 原因 | 解決策 |
|:---|:---|:---|:---|
| Bug-03 | 温度読取の一時的スパイク | SPI通信の一時乱れ | 最大3回リトライ処理を実装 |
| Bug-04 | SD書き込み後の読取値不定 | SPI共用バスの状態不定 | `SPI.end(); SPI.begin();` でバスリセット |
| Bug-05 | MicroSD初期化失敗 | CS ピン未指定（GPIO4が明示されていない） | `SD.begin(4)` でGPIO4に明示化 |

## 【課題解決の実績（Problem / Action / Result）】

### Case 1: 計測データの自動記録化

**Problem** 現場での温度評価時に、計測値の手書き記録またはExcel手入力が必要。評価1回あたりの後処理工数が大きい。  
**Action** MicroSD + FAT32 + CSVフォーマットを採用。10サンプル毎のバッファリング書き込みにより負荷を最適化。  
**Result** 計測終了時に DATA_NNNN.csv が自動生成。PCのExcelで直接読み込み・グラフ化が可能となり、手入力工数をゼロ化。

---

### Case 2: リアルタイム統計計算の実現

**Problem** 統計値（平均・標準偏差）が計測終了後の手計算または別ツール依存。計測中にばらつきのトレンドを把握できない。  
**Action** Welford法をLogic_Task（50ms周期）内で実装。配列蓄積なしの逐次更新により、メモリ増加を伴わずリアルタイム更新を実現。  
**Result** 計測中に平均・標準偏差・最大・最小が画面表示される。統計計算の外部依存を排除し、意思決定の即時性を確保。RAM使用率は計測時間に依存せず 7.2% で安定。

---

### Case 3: センサ信頼性の確保

**Problem** MAX31855からの読取値が稀に異常値（スパイク）を返す。CSVに混入すると統計値の信頼性が低下する。  
**Action** ローパスフィルタ（α=0.1）によるノイズ低減 + 異常値検出時に最大3回リトライ処理を実装。  
**Result** 計測精度 ±1℃ を安定維持。長時間計測においても外れ値による統計値の歪みを防止。

---

### Case 4: SPI共有バス競合の排除

**Problem** MicroSD書き込み後にMAX31855の読取値が不定になる。特定条件で計測値の信頼性が崩壊する（Bug-04）。  
**Action** SD書き込み完了後に `SPI.end(); SPI.begin();` を挿入してバス状態を強制リセット。  
**Result** SD書き込みとセンサ読取の全期間において干渉が生じなくなり、計測の継続安定性を確保。

---

### Case 5: リリース品質のテスト保証

**Problem** 複数バグが混在していたため、個別デバッグだけでは見落としリスクが高い。何をもって「完成」とするか基準が不明確。  
**Action** 7カテゴリ33ケースのテスト計画書を自作。テスト実行→問題発見→修正→再テストのループを計5回実施。  
**Result** 33/33（100%）合格を確認。引き継ぎ可能な「リリースレベル」の品質を客観的に証明。

## 【自己 PR ポイント（面接・エントリーシート用素材）】

### ポイント① ── 技術転用力（既存知識の応用）

PLC設計経験で習得した「スキャン周期」「工程歩進」「グローバルラベル」という制御工学の本質的な概念を、C++の「マルチレートタスク」「状態機械」「構造体変数管理」に体系的に対応させました。特定の言語・ツールへの依存ではなく、「制御の本質」として理解しているため、異なる技術環境でも同様のアプローチが適用できます。

### ポイント② ── 自走力（計画→実行→完遂）

16,082円の予算確保・部品調達・組み立て・ソフトウェア開発・33ケースのテスト・15ファイルのドキュメント整備まで、一切外部委託なしで完遂しました。「誰かに頼まれたから」ではなく、「課題を自分で発見して、自分で解決した」プロジェクトとして、自走型エンジニアとしての働き方を体現しています。

### ポイント③ ── デバッグ・問題解決力

5件の技術的バグを自力で根本原因まで追跡・修正しました。特に「Bug-01→Bug-02の思考連鎖」では、「統計値が全行0.0」という1つの症状の背後に2つの独立したバグが連鎖していることを段階的に解明しました。最初の修正後も症状が再現したとき、「別の原因が存在しているはず」と判断してコードを精読し、設計段階の計算タイミングバグを発見しました。

「症状が再現した＝同一バグの修正が不完全」という思い込みをせず、「症状が同じでも根本原因は別かもしれない」という視点を持って追跡した事例です。ロジックアナライザなし、シリアルモニタとCSVファイル目視という制約下での系統的診断力は、実際の開発現場でも応用可能なスキルです。

### ポイント④ ── ドキュメント文化（後工程を意識した開発）

「自分しかわからないコード」を作らないことを意識し、高校卒業1年目の後継者を想定した引き継ぎガイドを含む15ファイルのドキュメントを整備しました。「APIリファレンス」「トラブルシューティング手順書」「設計思想の解説書」「テスト計画書」という多角的なドキュメント体系は、チームでの開発でも通用するソフトウェアエンジニアリング実践の証拠です。
