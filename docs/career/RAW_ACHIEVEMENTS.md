### 1-4. アーキテクチャ詳細

**3タスク構成（マルチレート制御）**

| タスク名 | 実行周期 | 担当役割 |
|:---|:---|:---|
| IO_Task | 10ms | センサ読取（MAX31855）/ ローパスフィルタ / SD書き込み判定 |
| Logic_Task | 50ms | Welford法統計計算 / アラーム判定 / 状態遷移制御 |
| UI_Task | 200ms | LCD描画更新 / ボタン入力処理 |

**状態機械（State Machine）**
```
[IDLE] ──BtnA短押し──> [RUN] ──BtnA短押し──> [RESULT] ──BtnA短押し──> [IDLE]
  待機（センサ表示）       計測中（リアルタイム更新）   結果表示（統計値・CSV保存）
```

**アルゴリズム**

| アルゴリズム | 用途 | 特徴 |
|:---|:---|:---|
| Welford法 | 平均・標準偏差のオンライン計算 | データを蓄積せずにリアルタイムで統計値更新（メモリ効率◎） |
| 一次遅れフィルタ | センサノイズ低減 | α=0.1、前値90%＋新値10%で平滑化 |
| SDバッファリング | SD書き込み効率化 | 10サンプル毎に1行書き込み（都度書き込みより高速） |

**モジュール構成（6モジュール）**

| モジュール | ファイル | 責務 |
|:---|:---|:---|
| main | main.cpp | 初期化・メインループ・タスクスケジューリング |
| MeasurementCore | MeasurementCore.cpp/.h | 温度読取・Welford統計・フィルタ処理 |
| SDManager | SDManager.cpp/.h | CSV作成・バッファ書き込み・ファイル管理 |
| EEPROMManager | EEPROMManager.cpp/.h | HI/LOアラーム値の永続保存・読み出し |
| DisplayManager | DisplayManager.cpp/.h | LCD状態別描画（IDLE/RUN/RESULT画面） |
| IOController | IOController.cpp/.h | ボタン入力処理・状態遷移トリガー |

**CSV出力フォーマット（10列）**
```
ElapsedSec, Temp_C, State, Samples, Average_C, StdDev_C, Max_C, Min_C, HI_ALARM, LO_ALARM
```
- 改行コード: CRLF（Windows互換）
- ファイル命名: DATA_0000.csv〜DATA_9999.csv（自動インクリメント）
- 最大ファイル数: 10,000

## 2. 実施した作業（行動事実）

### 2-1. プロジェクト企画・初期設計

- 業務課題（ヒートガン/チャンバーの温度評価を手作業で行っていた）を特定
- M5Stack + K型熱電対という組み合わせを選定（コスト・実装難易度・拡張性のバランス）
- 16,082円でハードウェアを自ら調達・組み立て（ハンダ付け不要設計）
- PlatformIO + Arduino フレームワークの選定（ESP32での迅速な開発を優先）

### 2-2. ソフトウェア設計

- 3タスク分離アーキテクチャを設計（IO/Logic/UIの責務分離）
  - 設計根拠: センサ読取（10ms高頻度）とLCD描画（200ms低頻度）を干渉させない
- 状態機械（IDLE→RUN→RESULT→IDLE）を設計
  - 設計根拠: 「今何をしているか」を明確化し、意図しない動作を防ぐ
- グローバル構造体 `G` による全変数の一元管理を設計
  - 設計根拠: PLC的「グローバルラベル」の概念をC++に転用
- Welford法の採用判断（メモリ非蓄積型のオンライン統計計算）
  - 設計根拠: 長時間計測でも RAM 増加なし、リアルタイム統計更新が可能
- 6モジュールへの責務分割を設計・実装

### 2-3. ハードウェア組み立て・接続テスト

- K型熱電対 → MAX31855 → M5Stack のSPI接続を配線
- GPIO ピン割り当て（MAX31855 CS=GPIO5、MicroSD CS=GPIO4）を決定・実装
- M5Stack画面での動作確認（シリアルモニタ + LCD表示の二重確認）

### 2-4. デバッグ（5件の技術課題解決）

- Bug-01〜Bug-05（後述セクション3を参照）を自力で特定・修正
- ロジックアナライザなし、シリアルモニタとCSVファイル内容目視での診断
- 根本原因まで遡るデバッグ（「統計値が0.0」→「タイミング問題」→「タスク実行順序の設計ミス」）

### 2-5. テスト設計・実行

- 7カテゴリ33ケースのテスト計画書を自作（PHASE4_INTEGRATION_TEST_PLAN.md）
- テスト計画書に基づき全テストを実行・結果を記録
- テスト中に発見した問題を修正してから再テスト（TDD的ループ）

### 2-6. ドキュメント作成

- 15ファイルのドキュメント体系を設計・全文作成（総行数 数千行）

## 3. 解決した課題（問題解決）

### Bug-01: Average/StdDev が RUN 中すべて 0.0 のまま

| 項目 | 内容 |
|:---|:---|
| **発見方法** | CSVファイルの目視確認（Average_C列が全行0.0） |
| **症状** | Samples は正常にカウントアップ、だが Average_C = 0.0 が全行で続く |
| **仮説1** | 書き込み条件の問題 → `D_Count >= 2` で20〜30ms後に早期書き込み |
| **根本原因** | IO_Task（10ms）がLogic_Task（50ms）より先に書き込みを実行していた。Welford未実行のまま0.0が書き込まれていた |
| **解決策** | 書き込み条件を `D_Count >= 10`（100ms待機）に変更。Logic_Taskが最低2回完了を保証 |

### Bug-02: Welford計算は完了しているのにD_Averageが0.0のまま

| 項目 | 内容 |
|:---|:---|
| **発見方法** | Bug-01修正後の再テストで0.0継続を確認 → コード精読 |
| **症状** | D_Sum と D_M2 は更新されているが D_Average は NAN（→ 0.0 変換） |
| **根本原因** | D_Average = D_Sum / D_Count の除算が `handleButtonA()` の RESULT 遷移時のみに実装されていた設計バグ |
| **解決策** | Logic_Task（50ms周期）内で毎回 D_Average と D_StdDev を計算するよう変更 |

### Bug-03: 温度読取の一時的な異常値（スパイク）

| 項目 | 内容 |
|:---|:---|
| **症状** | MAX31855 の readCelsius() が稀に誤値を返す |
| **根本原因** | SPI通信の一時的な乱れ（SE値）によるもの |
| **解決策** | 異常値検出時に最大3回リトライを実装 |

### Bug-04: SD書き込み後の SPI バス競合

| 項目 | 内容 |
|:---|:---|
| **症状** | SD書き込み後に MAX31855 の読取値が不定になる現象 |
| **根本原因** | SD（SPI）書き込み後にSPIバスの状態が不定になり、MAX31855と共用するSPIバスに干渉 |
| **解決策** | SD書き込み後に `SPI.end(); SPI.begin();` でバスリセットを実装 |

### Bug-05: MicroSD 初期化失敗

| 項目 | 内容 |
|:---|:---|
| **症状** | 特定の条件でSDカードが初期化できない |
| **根本原因** | MicroSD の CS ピンが未指定（デフォルト動作）になっていたため、GPIO4 で初期化が成立しないケースがあった |
| **解決策** | `SD.begin(4)` として microSD CS = GPIO4 に明示化 |

## 4. 習得スキル（スキルマップ）

### 4-1. C++ 組み込みプログラミング

- **型安全設計**: `constexpr`による定数化、型サフィックスによる安全な型管理
- **構造体設計**: グローバル変数を `struct G` として一元管理
- **名前空間の活用**: モジュールを `namespace` で整理
- **マジックナンバー排除**: 全定数を `constexpr` で名前付き管理
- **エラーハンドリング**: NaN ガード、リトライ処理、モード別エラー表示
- **状態機械実装**: `enum class` + `switch` によるステートマシン

### 4-2. ESP32 マイコン開発

- **Arduino フレームワーク**: setup()/loop()、ハードウェア抽象化レイヤーの活用
- **マルチレート制御**: `millis()` タイマーによる非同期タスクスケジューリング
- **SPI 通信**: MAX31855（温度センサ）・MicroSD 同時接続、CS切り替え管理
- **I2C 通信**: EEPROM アクセス（設定データ永続化）
- **GPIO 制御**: ボタン入力のデバウンス処理、ピン設定

### 4-3. ハードウェア設計・組み立て

- **部品選定**: コスト（16,082円）・実装難易度・拡張性のバランスで最適選定
- **SPI バス設計**: 2デバイス共有時の CS ピン割り当て管理
- **ハンダ付け不要設計**: ジャンパワイヤ＋完成品モジュールによる容易な組み立て

### 4-4. アルゴリズム・数学

- **Welford 法**: オンライン（逐次更新型）平均・分散・標準偏差計算の実装
- **一次遅れフィルタ（ローパスフィルタ）**: ノイズ低減アルゴリズムの実装と係数チューニング
- **状態機械設計**: IDLE/RUN/RESULT の3状態遷移の設計と実装

### 4-5. ソフトウェア設計原則

- **関心の分離**: IO/Logic/UI の3タスクへの責務分離
- **モジュール化**: 6モジュールへのコード分割
- **保守性**: ドキュメントコメント、定数化、命名規則の統一
- **堅牢性**: エッジケース処理（ゼロ除算保護、NaNガード、タイムアウト処理）

### 4-6. 開発プロセス・ツール

- **PlatformIO**: ビルド・アップロード・ライブラリ管理
- **Git / GitHub**: バージョン管理、タグ管理（v1.0.0）、公開リポジトリ運用
- **テスト設計**: 7カテゴリ33ケースの統合テスト計画策定と実行
- **デバッグ手法**: シリアルモニタ＋CSVファイル content分析による非侵襲デバッグ
- **ドキュメント管理**: Markdown による15ファイルのドキュメント体系設計・運用
