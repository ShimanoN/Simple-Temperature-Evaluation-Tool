# PPTX Creation Workflow — AI Reference Guide

> **Purpose**: This document is the single source of truth for AI-assisted PPTX creation.  
> When a user says "I want to create a PPTX, refer to this file", the AI must follow
> this workflow precisely.
>
> **Language policy**: AI-internal logic and comments → English. All output content → Japanese.

---

## Overview

```
[Phase 1] AI interviews user → understands presentation purpose and structure
[Phase 2] AI drafts presentation_spec.md → user reviews and approves
[Phase 3] AI runs spec_to_pptx.py → PPTX is generated
[Phase 4] User reviews PPTX → AI iterates if needed
```

**Human responsibilities**: Decide content and approve spec  
**AI responsibilities**: Write spec.md, run converter, fix bugs, commit to Git

---

## Phase 1 — Spec Design Interview

When the user requests a new PPTX, AI must ask the following questions **in a single message**
(do not ask them one at a time). Group them naturally in Japanese.

### Required information to gather

| Category | Questions |
|----------|-----------|
| **Purpose** | What is this presentation for? (job interview, internal report, conference, etc.) |
| **Audience** | Who will see it? (recruiter, manager, engineers, clients) |
| **Scope** | What is the main message / key point to convey? |
| **Content** | What sections / topics should be covered? |
| **Hardware / Tech** | If technical: hardware model names, versions, costs (exact values) |
| **Slide count** | Rough target number of slides |

### AI behavior in Phase 1

1. Ask all questions in one message
2. Based on answers, **propose a slide structure** (numbered list with layout type)
3. Confirm with user before writing spec.md
4. If user approves, proceed to Phase 2

### Example slide structure proposal

```
1. タイトルスライド (layout: title)
2. 目次 (layout: toc)
3. プロジェクト概要 (layout: bullet)
4. コストメリット (layout: cost_comparison)
5. ハードウェア構成 (layout: two_column)
6. ピン割当 (layout: table)
7. ソフトウェア設計 (layout: bullet)
8. まとめ (layout: bullet)
```

## Phase 2 — Writing presentation_spec.md

### File location convention

```
docs/<project_name>/presentation_spec.md   ← spec file (human edits this)
docs/<project_name>/create_presentation.py ← auto-generated by AI, do not edit manually
docs/<project_name>/<output_name>.pptx     ← final output
```

### Spec file format

The spec file consists of two parts:

1. **YAML front matter** (between `---` delimiters): project-wide settings
2. **Slide sections**: one per slide, beginning with `## slide_NN | layout: LAYOUT_NAME`

```markdown
---
project: Simple Temperature Evaluation Tool
subtitle: M5Stack Basic V2.7 × MAX31855 熱電対温度計測システム
hardware_mcu: M5Stack Basic V2.7
hardware_sensor: MAX31855
tech_stack: ESP32  /  C++ (Arduino)  /  PlatformIO
version: v1.0.0  ｜  2026 年 3 月
output: Simple_Temp_Tool_Overview.pptx
---

## slide_01 | layout: title

## slide_02 | layout: toc
items:
  - "1. プロジェクト概要"
  - "2. ハードウェア"
  - "3. ソフトウェア設計"
  - "4. 実装詳細"

## slide_03 | layout: cost_comparison
title: コストメリット — 自作 vs. 市販品
section: Section 1  プロジェクト概要
cost_rows:
  - [M5Stack Basic V2.7, Amazon, "¥7,990"]
  - [MAX31855 ブレークアウト, 秋月電子, "¥980"]
  - [K型熱電対 (1m), Amazon 等, "¥700"]
  - [MicroSD (16GB), 量販店, "¥600"]
  - [ジャンパワイヤ, 秋月電子, "¥200"]
  - [合計, —, "¥9,570"]
market_rows:
  - [本ツール（自作）, M5Stack + MAX31855, "¥9,570"]
  - [標準製品, USB データロガー, "¥30,000〜¥50,000"]
  - [プロフェッショナル, 産業用データロガー, "¥100,000〜¥250,000+"]
highlight: 標準製品の 1/3.1 の価格で同等機能を実現
sub_highlight: "本ツール ¥9,570 vs 標準ロガー ¥30,000〜 → 最大 ¥20,000 以上削減"
checklist:
  - "✓ SD 自動記録  ✓ LCD リアルタイム表示  ✓ 上下限アラーム（ビープ音）"
  - "✓ スタンドアロン動作  ✓ EEPROM 設定保存  ✓ 統計解析（平均・σ・Max/Min）"
```

## Phase 3 — Running the Converter

```powershell
# From workspace root (Windows — use the project Python that has pptx + yaml):
& "C:\Users\島野拓実\AppData\Local\Python\bin\python.exe" tools/spec_to_pptx.py docs/<project_name>/presentation_spec.md

# Or if python is in PATH:
python tools/spec_to_pptx.py docs/<project_name>/presentation_spec.md
```

**Required Python packages**: `python-pptx`, `PyYAML`  
Install if missing: `pip install python-pptx pyyaml`

The converter reads the spec.md and outputs the PPTX at the path specified in `output`.

### AI responsibilities in Phase 3

1. Run the command in terminal
2. Verify the output message shows correct slide count
3. Check for any Python errors
4. If errors occur, fix them (do NOT ask user to fix Python)
5. Report success and file path to user

## Phase 4 — Review Loop

After PPTX is generated, the user reviews it.  
Common feedback types and how AI should handle them:

| Feedback | AI action |
|----------|-----------|
| "〇〇が重なっている" | Check `safe_y_after_table()` usage in that slide's renderer |
| "テキストが小さい" | Adjust `size=Pt(N)` in spec renderer |
| "型番が違う" | Update front matter `hardware_mcu` value — it propagates everywhere |
| "スライドを追加したい" | Add new `## slide_NN` section to spec.md, re-run |
| "レイアウトを変えたい" | Change `layout:` value in that slide section |

## Layout Reference

Each layout and its accepted YAML parameters are listed below.

---

### `layout: title`

Generates the title slide using project config from front matter.  
**No additional params needed.**

Front matter fields used:
- `project` → large title text
- `hardware_mcu` + `hardware_sensor` → subtitle (auto-formatted)
- `tech_stack` → tech stack line
- `version` → version/date line

---

### `layout: toc`

Table of contents slide.

```yaml
## slide_02 | layout: toc
title: 目次          # optional, default: "目次"
items:
  - "1. Section name"
  - "2. Section name"
```

---

### `layout: section_divider`

Section break slide with large section number and title.

```yaml
## slide_XX | layout: section_divider
section_num: "Section 2"
section_title: ハードウェア
description: M5Stack Basic V2.7 と MAX31855 の構成    # optional
```

---

### `layout: bullet`

General-purpose slide with title, bullet list, and optional highlight box.

```yaml
## slide_XX | layout: bullet
title: スライドタイトル
section: Section 1  プロジェクト概要    # header section label
bullets:
  - "箇条書き1"
  - "箇条書き2"
  - "箇条書き3"
highlight: "強調したいポイント"          # optional — navy box
sub_text: "補足説明テキスト"             # optional — below highlight
```

---

### `layout: table`

Slide with a data table and optional warning/note boxes below.

```yaml
## slide_XX | layout: table
title: SPI バス共有（LCD + MAX31855）
section: Section 2  ハードウェア
columns: [GPIO, 信号名, 接続先, 備考]
col_widths: [2.0, 2.5, 4.0, 4.1]        # inches, must sum to ~12.6
rows:
  - [GPIO 18, SCK (SPI Clock), MAX31855 CLK, LCD と共有]
  - [GPIO 19, MISO (SPI Data), MAX31855 DO, LCD と共有]
row_height: 0.45                          # optional, default: 0.40
warning: "⚠ SD カードの CS は GPIO4 を明示指定すること"     # optional — red warning box
warning_code: "SD.begin(TFCARD_CS_PIN, SPI, 40000000);"      # optional — code in warning
note: "EEPROM: ESP32 内蔵フラッシュを仮想 EEPROM として使用" # optional — gray note box
```

**IMPORTANT**: The converter automatically calculates `safe_y_after_table()` to place
warning and note boxes. Never manually adjust y-coordinates for these.

---

### `layout: cost_comparison`

Dual-table slide: cost breakdown (left) + market comparison (right) + highlight box + checklist.

```yaml
## slide_XX | layout: cost_comparison
title: コストメリット — 自作 vs. 市販品
section: Section 1  プロジェクト概要
cost_rows:                                           # left table (includes header row)
  - [部品, 購入先目安, 価格（税込）]                 # ← header row
  - [M5Stack Basic V2.7, Amazon, "¥7,990"]
  - [合計, —, "¥9,570"]
market_rows:                                         # right table (includes header row)
  - [カテゴリ, 代表製品, 概算価格]                   # ← header row
  - [本ツール（自作）, M5Stack + MAX31855, "¥9,570"]
  - [標準製品, USB データロガー, "¥30,000〜"]
highlight: "標準製品の 1/3.1 の価格で同等機能を実現"  # navy box title
sub_highlight: "¥9,570 vs ¥30,000〜 → 最大 ¥20,000 削減"   # orange subtext
checklist:                                            # optional gray box at bottom
  - "✓ 機能A  ✓ 機能B  ✓ 機能C"
  - "✓ 機能D  ✓ 機能E  ✓ 機能F"
```

---

### `layout: two_column`

Two-column layout for comparing or contrasting content.

```yaml
## slide_XX | layout: two_column
title: スライドタイトル
section: Section X  セクション名
left_title: 左カラムの見出し
left_items:
  - "左の項目1"
  - "左の項目2"
right_title: 右カラムの見出し
right_items:
  - "右の項目1"
  - "右の項目2"
note: "下部の補足テキスト"    # optional
```

## Coordinate System Reference

```
(0, 0) ──────────────── (13.33", 0)   top-left origin
  │  Header bar (h=1.0")              │
  │                                   │
  │  Content area                     │  CX=0.35"  CY=1.1"  CW=12.63"
  │  y: 1.1" → 6.85" (safe limit)    │
  │                                   │
  │  Footer bar (y=6.9")              │
(0, 7.5") ──────────── (13.33", 7.5") bottom-right
```

### Table bottom calculation

$$\text{table\_end\_y} = \text{start\_y} + \text{num\_rows} \times \text{row\_height}$$

The converter calls `safe_y_after_table()` automatically.  
**Spec authors never need to calculate y-coordinates manually.**

## AI Behavior Checklist

When starting a new PPTX session, AI must:

- [ ] Read this file completely before responding
- [ ] Ask all Phase 1 questions in a single message
- [ ] Propose slide structure and wait for approval before writing spec.md
- [ ] Write spec.md using only documented layouts and parameters
- [ ] Verify `col_widths` sum equals approximately 12.6" for full-width tables
- [ ] Run `python tools/spec_to_pptx.py <spec_path>` and report result
- [ ] Never ask user to edit Python files — AI handles all code

---

## Common Mistakes to Avoid

| Mistake | Prevention |
|---------|-----------|
| Hardware name inconsistency (e.g. Core2 vs Basic V2.7) | Always use `hardware_mcu` from front matter — it propagates to all slides |
| Table overlapping with blocks below | Converter uses `safe_y_after_table()` automatically |
| Content overflowing footer | Converter checks `CONTENT_BOTTOM = 6.85"` limit |
| YAML parsing errors | Wrap values containing special chars (¥, :, -) in double quotes |
| PowerShell git commit with ¥ in message | Use single quotes for commit messages or ASCII only |
